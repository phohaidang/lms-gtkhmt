BUỔI 5 — LỚP, ĐỐI TƯỢNG & LẬP TRÌNH HƯỚNG ĐỐI TƯỢNG

Chương 5 của giáo trình · Thời lượng 5 giờ trên lớp. Vai trò trong câu chuyện xuyên suốt: Buổi 4 ta đã có hàm xử lý 1 khách hàng. Nhưng khi quản lý 1000 khách, code thành "rừng dictionary và list rối rắm". Buổi này ta nâng cấp lên kiến trúc OOP — mỗi khách hàng là một đối tượng với dữ liệu và hành vi riêng.

Trọng tâm đề thi: Chương 5 chiếm 9/40 câu (22.5%) — cao thứ 2 sau Chương 6. Đây là buổi không thể bỏ qua.

I. MỤC TIÊU BUỔI HỌC

Sau buổi này, sinh viên có thể:

  1. Phân biệt lập trình thủ tục (procedural) và lập trình hướng đối tượng (OOP).
  2. Định nghĩa lớp (class), tạo đối tượng (object/instance) trong Python.
  3. Sử dụng __init__ để khởi tạo đối tượng.
  4. Phân biệt thuộc tính (attribute) và phương thức (method).
  5. Sử dụng self đúng cách.
  6. Xây dựng quan hệ kế thừa (inheritance) giữa lớp cha và lớp con.
  7. Giải thích đơn giản các nguyên lý: encapsulation (đóng gói), inheritance (kế thừa), polymorphism (đa hình).

Kết nối CLO: CLO1, CLO2, CLO3.

Kết nối đề thi: Đề mẫu 02 chuyên về OOP — nên xem trước với SV.

Phần Nội dung
Situation Cuối Buổi 4, ta quản lý nhiều khách hàng bằng cách dùng list của dictionary: khach[0] = {"ten": ..., "so_du": ..., "lich_su": [...]}. Mỗi khách có 10 thuộc tính, có 5 thao tác (nạp, rút, xem, đóng, in báo cáo).
Complication Code trở thành "rừng": khach[i]["lich_su"].append(...), if khach[i]["loai"] == "VIP" ... Mỗi lần thêm tính năng mới, ta lại sửa khắp nơi. Nếu có 2 loại tài khoản (vãng lai vs tiết kiệm) với cách tính lãi khác nhau — ta sẽ viết thế nào?
Question Làm sao gắn dữ liệu với hành vi thành một thể thống nhất? Làm sao tái sử dụng cho 2 loại tài khoản mà không copy-paste?
Answer Class = bản thiết kế. Object = sản phẩm tạo ra từ thiết kế. Mỗi object mang theo cả dữ liệu và hành vi của riêng nó. Kế thừa cho phép loại con tái sử dụng từ loại cha.

III. KHỞI ĐỘNG TỪ "CÁI BIẾT"

3 ẩn dụ chuẩn:

  1. Khuôn bánh và cái bánh. Khuôn = class, bánh = object. Một khuôn làm ra nhiều bánh giống cấu trúc nhưng khác nhau (có cái nho, có cái sô-cô-la).
  2. Chứng minh thư. Mọi người Việt đều có CMND/CCCD theo cùng một mẫu (class), nhưng mỗi người có nội dung riêng (object). Bản thân CCCD có hành vi: "kiểm tra hết hạn", "in lại"...
  3. App điện thoại. App Vietcombank và app Techcombank đều là "app ngân hàng" (lớp cha), kế thừa các tính năng chung (đăng nhập, chuyển khoản), nhưng mỗi app có giao diện và tính năng riêng.

    def rut(self, so): if so <= self.so_du: self.so_du -= so print(f"Đã rút {so}. Số dư: {self.so_du}") else: print("Số dư không đủ.")

Hoạt động 3.2 (35 phút) — Cú pháp kế thừa

  • Lớp cha: ```python class TaiKhoan: def init(self, ten, so_du): self.ten = ten self.so_du = so_du

    def nap(self, so): self.so_du += so

    def rut(self, so): if so <= self.so_du: self.so_du -= so return True return False

    def str(self): return f"TK[{self.ten}]: {self.so_du:,}đ" ```

  • Lớp con — Tài khoản tiết kiệm: ```python class TKTietKiem(TaiKhoan): def init(self, ten, so_du, lai_suat): super().init(ten, so_du) # gọi init của cha self.lai_suat = lai_suat

    def cong_lai(self): self.so_du *= (1 + self.lai_suat / 100)

    # rút có phạt def rut(self, so): phat = so * 0.01 # phạt 1% nếu rút trước hạn tong = so + phat if tong <= self.so_du: self.so_du -= tong return True return False ```

  • Demo override: ```python vl = TaiKhoan("Lan", 100_000) tk = TKTietKiem("Nam", 100_000, 7)

vl.rut(50_000) # còn 50,000 tk.rut(50_000) # còn 49,500 (mất phạt 500) tk.cong_lai() # tăng lãi ```

  • Khái niệm cần nhấn mạnh:
  • super().__init__(...) — gọi constructor cha.
  • Override: lớp con định nghĩa lại phương thức của cha.
  • Polymorphism: cùng tên rut, hành vi khác nhau ở 2 lớp.
